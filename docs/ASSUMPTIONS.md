# Assumptions

This document records all assumptions made during the design and implementation of the Digital Government Service Request Platform.

---

## Authentication & User Management

1. **Email as username.** A user's email address is used as their login username. Email is unique across all users in the system.

2. **Citizens are created by admins only.** Citizens cannot self-register. An admin creates a citizen account and provides a temporary password. The citizen must change their password on first login.

3. **Password complexity is validated at the API layer.** The following rules are enforced by bean validation annotations on `ChangePasswordRequest.newPassword` and `CreateCitizenRequest.temporaryPassword`:
   - Minimum 8 characters (`@Size(min = 8)`)
   - At least one uppercase letter (`@Pattern`)
   - At least one digit (`@Pattern`)
   - At least one special character — any non-alphanumeric character (`@Pattern`)

   The `LoginRequest.password` field has no complexity check (it accepts any non-blank value, since validating complexity at login would lock out users with legacy passwords).

4. **Single active session per user is enforced via token versioning.** Each user has a `token_version` counter in the database (default 0). On every successful login, the counter is incremented and its value is embedded as a `ver` claim in the issued JWT. On every authenticated request, the JWT filter compares the `ver` claim against the current database value — if they do not match, the token is rejected (401) and the client is redirected to login. This means a second login from any device immediately invalidates all previously issued tokens for that user. No refresh token mechanism is implemented; users must re-authenticate after their token expires.

---

## Citizen Management

5. **NIC uniqueness is system-wide.** No two citizens can share the same NIC. This is enforced at both the application layer (check before insert) and the database layer (unique constraint).

6. **Deactivation is a soft delete.** Deactivating a citizen sets the citizen's status to `INACTIVE` and disables the linked user account (`active = false`). All historical data (service requests, documents, notifications) is retained. There is no hard-delete endpoint for citizens.

7. **Citizen reference format.** Citizen references follow the format `CIT-XXXXXXXX` where `X` is an alphanumeric character generated by `ReferenceGenerator`. Uniqueness is verified before saving.

8. **Citizens can only view and act on their own data.** A citizen cannot view or modify another citizen's service requests, documents, or notifications. This is enforced at the service layer by comparing the authenticated user's username against the citizen profile's linked user.

---

## Service Requests

9. **Service type is a controlled list served from the API.** The database column is `VARCHAR(100)` with no DB-level enum constraint, but the application enforces a fixed list: `PASSPORT_RENEWAL`, `BIRTH_CERTIFICATE`, `DRIVING_LICENSE`, `BUSINESS_REGISTRATION`, `OTHER`. The list is exposed via `GET /v1/service-requests/types` and consumed by the frontend dropdowns (citizen request creation and agent request edit). Adding a new service type requires only a code change to the controller list — no schema migration.

10. **Only one active request per flow.** No business rule limits how many open service requests a citizen can have simultaneously. Multiple concurrent requests are allowed.

11. **Status transitions are strictly enforced.** The transition graph is:
    - `SUBMITTED` → `IN_REVIEW` (by SERVICE_AGENT)
    - `IN_REVIEW` → `APPROVED` or `REJECTED` (by SERVICE_AGENT)
    - `SUBMITTED` or `IN_REVIEW` → `CANCELLED` (by ADMIN only)
    - `APPROVED`, `REJECTED`, and `CANCELLED` are terminal — no further transitions are permitted.

12. **Request reference format.** Request references follow the format `REQ-XXXXXXXX`, generated the same way as citizen references.

---

## Supporting Documents

13. **Document storage is external.** The platform stores only document metadata (file name, size, MIME type, verification status) and an opaque `storageReference` string. Actual file upload and retrieval are the responsibility of an external storage system (e.g., object storage). No file upload endpoint is implemented.

14. **Document ownership follows the service request.** A document belongs to the citizen who owns the parent service request. A citizen cannot add a document to another citizen's request.

15. **Documents can be hard-deleted.** Unlike citizens and service requests, document metadata records can be permanently deleted via the DELETE endpoint.

16. **Document reference format.** Document references follow the format `DOC-XXXXXXXX`.

---

## Notifications

17. **Notifications are generated on status change only.** A notification is created every time a service request status changes (e.g., SUBMITTED → IN_REVIEW, IN_REVIEW → APPROVED). No notification is created at the moment of initial submission (the first status history entry has no prior status).

18. **Notifications are read-only for citizens.** Citizens can view and mark notifications as read. They cannot delete notifications.

19. **Notifications are per-citizen.** Each notification is linked to the citizen who owns the service request. Admins and agents do not receive notifications.

---

## Status History

20. **Status history is append-only.** Records are never updated or deleted. This provides a complete, tamper-evident audit trail of every status transition.

21. **The initial SUBMITTED entry has a null `old_status`.** When a service request is created, the first history record is inserted with `old_status = NULL` and `new_status = 'SUBMITTED'`.

22. **`changed_by` stores the username.** The username (email) of the authenticated user who triggered the transition is recorded in `status_history.changed_by`.

---

## Infrastructure & Deployment

23. **Docker Compose is the primary deployment method.** The application is designed to run via `docker compose up --build`. Local development without Docker is also supported if Java 17+ and PostgreSQL are available.

24. **Single-instance deployment.** The application is designed for a single-instance deployment. No clustering, load balancing, or distributed caching is implemented.

25. **Database connection uses environment variables.** `DB_URL`, `DB_USERNAME`, `DB_PASSWORD`, `JWT_SECRET`, and `JWT_EXPIRATION_MS` are injected via environment variables. Sensible local defaults are provided in `application.yml` for development convenience.

26. **Flyway manages all schema changes.** No manual DDL is required. On startup, Flyway applies all pending migrations. The initial schema is created by `V1__create_initial_schema.sql`.

---

## Security

27. **HTTPS is assumed at the infrastructure level.** The application itself does not enforce TLS. In a production deployment, TLS termination is expected to be handled by a reverse proxy or load balancer in front of the application.

28. **JWT secret must be externally provided in production.** The `application.yml` does not provide a default JWT secret. It must be supplied via the `JWT_SECRET` environment variable. For the Docker Compose setup, a placeholder value is included in `docker-compose.yml` for local development only.

29. **CORS is configured.** Allow traffic from front end react app (localhost:3000).

30. **CSRF protection is disabled.** Since the API is stateless (JWT-based, no cookies), CSRF attacks are not applicable. CSRF protection is explicitly disabled in `SecurityConfig`.
